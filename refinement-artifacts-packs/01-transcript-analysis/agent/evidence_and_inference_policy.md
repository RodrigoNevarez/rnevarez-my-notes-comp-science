# Evidence and Inference Policy

## Evidence rules
- Use short excerpts only.
- Include 2–4 excerpts per important component.
- Prefer excerpts that support:
  - why the component matters
  - a key relationship
  - a dependency or constraint
  - a notable uncertainty
- Keep evidence in a separate **Evidence** subsection.
- Do not overload the report with large transcript blocks.

## Inference rules
- Inference is allowed when the transcript strongly implies a relationship, role, dependency, or significance.
- Every inference should be clearly labeled as inference.
- Inference should extend the transcript carefully, not invent beyond it.
- If evidence is weak, state the ambiguity instead of inferring aggressively.

## Acceptable inference examples
- A relationship is implied through repeated coordination language.
- A component appears upstream or downstream based on how speakers describe behavior.
- A person appears to own a domain because they repeatedly explain it or answer for it.

## Unacceptable inference examples
- Stating unconfirmed architecture as fact
- Assigning ownership with no supporting signal
- Creating dependencies not grounded in the transcript
- Treating a passing mention as a core system element

## Tie-break rule
When unsure whether to infer or hold back, prefer:
1. explicit statement
2. careful inference with clear labeling
3. unresolved ambiguity

Never skip directly from weak evidence to confident conclusion.
