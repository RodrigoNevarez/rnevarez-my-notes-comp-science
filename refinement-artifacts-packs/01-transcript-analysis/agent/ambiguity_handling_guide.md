# Ambiguity Handling Guide

## Principle
Ambiguity is not a failure of the report. It is part of the value of the report.

## When to surface ambiguity
Surface ambiguity when it blocks:
- architectural understanding
- task relevance
- dependency reasoning
- ownership follow-up
- next-step planning

## Common ambiguity types
- an important component is named but not explained
- a relationship is implied but not stated clearly
- ownership is hinted at but not explicit
- a decision is referenced without enough background
- a dependency is mentioned without describing impact

## How to write ambiguity well
- say what appears to be true
- say what is still unclear
- say why the missing context matters
- suggest what to investigate next

## Avoid
- pretending the transcript was complete
- replacing missing context with overconfident inference
- listing every ambiguity regardless of relevance

## Good ambiguity outputs
- “X appears central to the flow, but the transcript never explains whether it is upstream of Y or simply adjacent to it.”
- “Z is referenced as if it is already known, which suggests it may be important, but the transcript does not define its role.”
